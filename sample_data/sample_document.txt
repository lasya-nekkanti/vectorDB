Introduction to Vector Databases

A vector database is a specialized storage system designed to store, index, and search high-dimensional vectors efficiently. These vectors are typically generated by machine learning models such as embedding models, which convert text, images, or audio into numerical representations. Unlike traditional relational databases that search by exact matches, vector databases search by similarity, finding items whose vectors are closest to a given query vector in high-dimensional space.

The rise of large language models and retrieval-augmented generation has made vector databases a critical piece of modern AI infrastructure. When an application needs to find semantically similar content, such as relevant documents for a chatbot to reference, a vector database can perform this search in milliseconds, even across millions of stored vectors.

How Embeddings Work

Embedding models take a piece of text and convert it into a fixed-length array of numbers, called a vector. Sentences with similar meanings produce vectors that are close together in this numerical space, even if they use completely different words. For example, "the cat sat on the mat" and "a feline rested on the rug" would produce vectors that are close to each other, despite sharing almost no vocabulary.

This property is what makes semantic search possible. Instead of matching keywords, the system compares the meaning of the query to the meaning of stored content. Popular embedding models include OpenAI's text-embedding models, Google's models, and open-source alternatives like the sentence-transformers library, which includes models such as all-MiniLM-L6-v2.

Chunking Strategies

Before text can be embedded, it usually needs to be split into smaller pieces called chunks. This is necessary for several reasons. First, embedding models have a maximum input length and cannot process very long documents in a single pass. Second, smaller chunks produce more precise search results, since a giant chunk covering many topics will have a less focused embedding than a small, topic-specific one.

There are several common chunking strategies. Fixed-size chunking splits text into chunks of a set number of characters or tokens, which is simple but can cut sentences awkwardly. Sentence-based chunking splits text along sentence boundaries, preserving semantic completeness. Recursive chunking tries to split along natural boundaries like paragraphs first, falling back to sentences or characters only when a chunk is still too large. Overlapping chunks, where a few sentences are repeated between consecutive chunks, help preserve context across chunk boundaries.

Choosing the right chunk size involves tradeoffs. Smaller chunks give more precise retrieval but may lack surrounding context. Larger chunks preserve more context but can dilute the embedding's focus and may exceed model input limits. A common starting point for many applications is chunks of three to eight sentences with a one or two sentence overlap.

Introduction to ChromaDB

ChromaDB is an open-source vector database designed to be simple to set up and use, particularly for local development and small to medium scale applications. It can run entirely in-process within a Python application, persisting data to a local directory on disk. This makes it an excellent choice for prototypes, student projects, and applications that do not need a separately hosted database server.

ChromaDB stores three things for each entry: the vector embedding itself, the original text it was generated from, and optional metadata such as the source document name or chunk index. This metadata is extremely useful when you want to trace a search result back to its original source.

Querying a ChromaDB collection involves embedding the search query using the same model that was used to embed the stored documents, then asking the collection for the nearest neighbors by distance. ChromaDB supports several distance metrics, including cosine similarity, L2 (Euclidean) distance, and inner product, with cosine similarity being the most common default for text embeddings.

Alternative Vector Database Options

While ChromaDB is popular for local projects, several other vector databases serve different needs. FAISS, developed by Meta, is a library rather than a full database; it is extremely fast and lightweight but lacks built-in persistence and metadata filtering out of the box, requiring more manual setup.

Pinecone is a fully managed, cloud-hosted vector database designed for production-scale applications. It offers automatic scaling, high availability, and a generous free tier, but requires an internet connection and an API key, unlike ChromaDB which can run fully offline.

Weaviate and Qdrant are other open-source vector databases that, like ChromaDB, can be self-hosted but also offer cloud-hosted versions for teams that need to scale beyond a single machine. Milvus is designed specifically for very large scale deployments, often used by companies storing billions of vectors.

Building a Document Ingestion Pipeline

A typical document-to-vector-database pipeline consists of four main stages. The first stage is loading, where the raw document, whether a PDF, Word document, or plain text file, is converted into a single string of text. Different file formats require different extraction libraries.

The second stage is chunking, where the raw text is split into smaller, semantically coherent pieces, as discussed earlier. The third stage is embedding, where each chunk is passed through an embedding model to generate its vector representation. The fourth and final stage is storage, where the chunks, their embeddings, and any relevant metadata are saved into the vector database.

Once this pipeline has been run, the vector database can be queried at any time without needing to repeat the loading, chunking, or embedding steps, since all of that work has already been done and persisted to disk.

Use Cases for Vector Databases

Vector databases power a wide range of applications. Retrieval-augmented generation, often abbreviated as RAG, is one of the most common use cases, where a language model retrieves relevant chunks of text from a vector database before generating an answer, grounding its response in real source material rather than relying solely on what it learned during training.

Semantic search engines use vector databases to let users search by meaning rather than exact keyword matches, improving results for queries phrased differently than the underlying content. Recommendation systems use vector similarity to find items, products, or content similar to what a user has previously liked or viewed.

Duplicate detection and clustering are also common applications, where vector databases help identify near-duplicate documents or group similar items together based on the distance between their embeddings.

Evaluating Retrieval Quality

When building a retrieval system, it is important to evaluate whether the chunks being returned are actually relevant to the query. Common metrics include precision at k, which measures what fraction of the top k retrieved chunks are actually relevant, and recall, which measures what fraction of all relevant chunks in the database were successfully retrieved.

Manual evaluation, where a developer reviews query results by hand, is also valuable during development, especially for small projects, since it can reveal chunking or embedding issues that automated metrics might miss. Common problems include chunks that are too large and dilute relevance, chunks that are too small and lack context, or an embedding model that is not well suited to the domain of the text being searched.

Conclusion

Converting a document into a vector database is a foundational skill for building modern AI-powered search and retrieval applications. The process involves loading text from a source document, splitting it into well-formed chunks, generating embeddings for each chunk using a model like sentence-transformers, and storing the results in a vector database such as ChromaDB. Once built, this vector database can answer natural language queries by finding the most semantically similar chunks, enabling powerful applications like semantic search and retrieval-augmented generation.
